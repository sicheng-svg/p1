### Q1：点赞与踩的互斥策略

**问题**：用户点赞了再踩，你怎么处理？

**你的原始回答**：静默忽略，直接返回。想法是以第二次操作为准，自动取消前一种。

**问题分析**：
- 代码中实际实现是静默返回 nil，并没有实现"自动取消"的逻辑，属于功能缺失
- `count += 0` 是为了避免 Go 编译器报"变量未使用"的错误，但这不是正确做法

**优化后的回答**：

> 业界主流做法（B站、YouTube、抖音）是"以最新操作为准"——如果用户已经点赞了，再点踩，就自动取消点赞并执行踩。具体实现：
>
> 1. 检测到用户对同一目标已存在相反操作
> 2. 先调用 CancelLike 取消旧操作
> 3. 再执行 Hate 新操作
> 4. 返回给客户端明确的状态变更信息，让前端同步更新 UI
>
> 对于不需要的返回值，Go 的惯用写法是用 `_` 忽略：
> ```go
> check, _, err := h.repo.IsUserAlreadyAction(req.TargetUid, common.HATE, req.Uid)
> ```

**面试要点**：
- 不要静默忽略，用户无感知是不好的体验
- 说方案时要说清楚你**已经实现了**还是**计划实现**，不要混淆

### Q2：并发安全 — check-then-act 的竞态问题

**问题**：同一用户同一时刻发了两次点赞，先查后写的流程会不会出问题？

**你的原始回答**：确实存在，可以在查之后写之前加锁。不能加 Go 层的锁，会导致并发量大幅下降。

**问题分析**：
- 意识到了问题的存在，方向正确
- 但排除 Go 本地锁的理由不够准确——核心问题不是性能，而是**多实例部署下不生效**
- 对锁粒度没有概念

**优化后的回答**：

> 会出问题。两个请求都可能通过校验，导致重复写入（比如 fans 数组里同一个 uid 出现两次）。解决方案是加 **Redis 分布式锁**。
>
> **为什么不用 Go 的 sync.Mutex**：
> 它只对单进程有效。如果服务部署了多个实例（生产环境的常态），A 实例上的 Mutex 管不住 B 实例上的请求。所以核心问题是**分布式场景下不生效**，不仅仅是性能问题。
>
> **为什么不建议用数据库锁**：
> MongoDB 没有传统 RDBMS 的行级悲观锁。虽然可以用 findAndModify 做原子操作，但在复杂的多步业务逻辑中不够灵活。
>
> **锁的粒度设计**：
>
> | 粒度 | 锁的 key | 效果 |
> |------|----------|------|
> | 全局锁 | `lock:like` | 所有用户排队，并发归零，**绝对不能用** |
> | 按用户 | `lock:user:{userId}` | 该用户所有操作串行，不同用户互不影响 |
> | 按用户+目标 | `lock:like:{userId}:{targetId}` | **粒度最细，推荐**。只有同一用户对同一目标串行 |
>
> 锁粒度越细，能并行的请求越多。我们要防的是"同一人对同一目标的重复操作"，所以锁到用户+目标这个粒度就够了。
>
> Redis 分布式锁的基本实现：
> ```
> SET lock:like:{userId}:{targetId} {requestId} NX PX 3000
> ```
> - NX：只在 key 不存在时设置（互斥）
> - PX 3000：3 秒自动过期（防死锁）
> - requestId：释放锁时校验身份，防止误删别人的锁

### Q3：热点数据阈值设计

**问题**：为什么选 1000 作为热点阈值？这个数字是怎么来的？

**你的原始回答**：领导定的。可以根据最近几天的访问量趋势来判断。

**问题分析**：
- "领导定的"在面试中是减分项
- "访问量趋势"的思路是动态热点发现，但和代码中按粉丝数量判断的实现不一致

**优化后的回答**：

> 1000 是上线初期的经验值，实际应该结合**线上监控数据动态调整**：
>
> 1. **监控驱动调整**：观察 MongoDB 的查询延迟 P99，当某些文档的查询延迟明显升高，说明阈值可能需要下调
> 2. **认识到静态阈值的局限性**：
>    - "粉丝多 ≠ 访问频繁"——一个半年前的热帖粉丝10万但已无人访问
>    - "粉丝少 ≠ 不热"——一个新帖正在被疯狂点赞但粉丝还不到1000
> 3. **理想方案**：用**滑动窗口统计 QPS** 做动态热点发现。比如最近5分钟内某个目标的访问次数超过阈值，就标记为热点迁移到 Redis；持续一段时间没有访问就降级回 MongoDB
>
> 但在项目初期，静态阈值是最简单可行的方案，先上线再迭代优化。

### Q4：Redis 与 MongoDB 双写一致性

**问题**：Redis 写成功、MongoDB 写失败怎么办？反过来呢？

**你的原始回答**：短暂不同步可以接受。可以先写 MongoDB，Redis 异步同步过来。

**问题分析**：
- "短暂不一致可以接受"这个判断非常好，说明你能根据业务场景做取舍
- 但"先写 MongoDB 再异步同步 Redis"这个方案与你"读请求优先读 Redis"的设计存在矛盾

**优化后的回答**：

> 对于点赞这种场景，不需要强一致，**最终一致**就够了。这不是金融转账，少显示一个赞不会造成资损。
>
> 三种方案的对比：
>
> | 方案 | 优点 | 缺点 | 适用场景 |
> |------|------|------|----------|
> | 先写MongoDB，异步同步Redis | 数据持久化有保障，不丢 | 读请求可能读到旧数据（因为优先读Redis） | 对数据安全要求高 |
> | 先写Redis，异步写MongoDB | 读取实时性好 | Redis宕机可能丢数据 | 对实时性要求高 |
> | 同步双写（我的做法） | 实时性好，不易丢 | 存在半成功问题 | 折中方案 |
>
> 我选择同步双写，针对半成功的情况，通过以下机制保证最终一致：
> 1. **失败重试**：写 Redis 失败后放入消息队列延迟重试
> 2. **定期对账**：后台任务定期比对 Redis 和 MongoDB 的数据，修复不一致
> 3. **以 MongoDB 为准**：Redis 本质是缓存，出现不一致时以 MongoDB 为最终数据源

### Q5：MongoDB 大文档膨胀问题

**问题**：100 万人点赞，fans 数组会不会撑爆 16MB 文档限制？

**你的原始回答**：确实有问题，不知道怎么解决。

**优化后的回答**：

> 会的，MongoDB 单文档上限 **16MB**。假设每条粉丝记录约 50 字节（uid + count + bson开销），100 万条就是约 50MB，远超限制。而且即使没超限制，**数组越大，$push/$pull 操作越慢**，因为 MongoDB 可能需要重新分配和移动整个文档。
>
> 解决方案有两种：
>
> **方案一：分桶（Bucketing）**
> ```
> { target_uid: "456", action: 1, bucket: 0, fans: [... 共1000个] }
> { target_uid: "456", action: 1, bucket: 1, fans: [... 共1000个] }
> ```
> 每个桶固定存 1000 个粉丝，满了创建新桶。索引建在 `(target_uid, action, bucket)` 上。
> - 优点：单桶内的批量查询效率高
> - 缺点：查某个 uid 是否存在需要扫描多个桶（除非额外加索引）
>
> **方案二：独立文档（推荐）**
> ```
> { target_uid: "456", action: 1, fan_uid: "1001", count: 1 }
> { target_uid: "456", action: 1, fan_uid: "1002", count: 3 }
> ```
> 每条点赞关系是一个独立文档，给 `(target_uid, action, fan_uid)` 建**唯一复合索引**。
> - 优点：查询、更新都是精确命中单文档，性能稳定，无膨胀风险
> - 缺点：文档数量变多，但 MongoDB 本身就擅长处理海量文档
> - 额外好处：唯一索引天然防止重复点赞，数据库层面的兜底保障

### Q6：为什么用 Redis ZSet 而不是 Hash

**问题**：热点数据存 Redis，为什么选 ZSet？

**你的原始回答**：不太了解，领导让用的。

**优化后的回答**：

> ZSet 的 **member 存 fan_uid，score 存 count（点赞次数）**。选 ZSet 的原因：
>
> | 操作 | ZSet | Hash |
> |------|------|------|
> | 查某用户是否点赞过 | ZSCORE O(1) | HEXISTS O(1) |
> | 查总点赞人数 | ZCARD O(1) | HLEN O(1) |
> | 查点赞最多的 TOP N | ZREVRANGE O(logN+M) **天然支持** | 需要 HGETALL 全取出再排序 O(N) |
> | 按分数范围查询 | ZRANGEBYSCORE **天然支持** | 不支持 |
>
> 基本的 KV 操作两者差不多，但 ZSet 的**排序和范围查询能力**是 Hash 不具备的。如果后续要做"谁给这个帖子点赞最多"的排行榜需求，ZSet 直接就能支撑，Hash 就得大改。
>
> **技术选型的核心原则**：在性能差不多的前提下，选扩展性更好的方案。

### Q7：迁移过程中的数据丢失

**问题**：粉丝数从 999 涨到 1001 触发迁移，迁移过程中新来的点赞会丢吗？

**你的原始回答**：确实没考虑过，应该会导致吞数据，少加了几个点赞。

**问题分析**：
- 能意识到会丢数据，分析正确

**优化后的回答**：

> 会丢。具体过程：
> 1. 迁移任务从 MongoDB 读出 fans 快照（假设此时有1001条）
> 2. 迁移任务开始逐条写入 Redis
> 3. **此时来了一个新的点赞请求**，由于 Redis 中还没有完整数据，isHotData 判断为"不是热点"，只写了 MongoDB
> 4. 迁移任务完成，Redis 里是旧的快照，缺少了第3步的那条点赞
> 5. 后续读请求优先查 Redis，这条数据就丢了
>
> **解决方案**：设置**迁移状态标记**
> 1. 迁移开始前，在 Redis 中设置 `migrating:{target_uid}` 标记
> 2. 迁移期间，所有写请求看到这个标记，就**同时往 Redis 和 MongoDB 都写**（即使 Redis 中还在迁移）
> 3. 迁移完成后，删除标记，恢复正常流程
>
> 这本质上是一种**双写过渡期**策略，类似于数据库迁移中的"灰度切换"。

### Q8：如何高效统计总赞数

**问题**：查"某个帖子一共多少赞"，怎么做？

**你的原始回答**：给文档加 count 字段，每次点赞更新。

**问题分析**：
- 方向正确，但需要关注原子性

**优化后的回答**：

> 在文档中增加 `total_likes` 计数字段。关键是要和 fans 的更新放在**同一个 UpdateOne 操作**里：
>
> ```js
> {
>   "$push": { "fans": { "uid": "123", "count": 1 } },
>   "$inc":  { "total_likes": 1 }
> }
> ```
>
> **为什么不用事务**：MongoDB 保证**单文档操作的原子性**——一个 UpdateOne 里的多个操作要么全成功要么全失败，不需要事务。
>
> **什么时候才需要事务**：跨文档、跨集合的操作才需要。比如"从用户A的积分文档扣分，同时往用户B的积分文档加分"，这是两个文档，必须用事务。
>
> 这是 MongoDB 的高频面试点：**单文档原子性 vs 多文档事务**，一定要区分清楚。

### Q9：系统如何扩展到 50 万 QPS

**问题**：QPS 从 1000 涨到 50 万，怎么扩展？

**你的原始回答**：切换为分布式，通过负载均衡器均衡分配。

**问题分析**：
- 方向对，但太笼统。只提到了应用层，没有考虑数据层瓶颈

**优化后的回答**：

> 全链路扩容，按层分析：
>
> | 层级 | 瓶颈 | 解决手段 |
> |------|------|----------|
> | 接入层 | 单点网关吞吐上限 | Nginx/网关集群 + 负载均衡 |
> | 应用层 | 单实例处理能力 | 无状态服务水平扩展（加实例） |
> | 缓存层 | 单 Redis 实例内存和连接数上限 | Redis Cluster 分片，按 target_id 哈希分到不同节点 |
> | 数据库层 | MongoDB 单机写入上限 | MongoDB 分片集群，target_id 作为 sharding key |
> | 异步削峰 | 突发流量直接打到数据库 | 引入消息队列（Kafka），点赞先入队，消费者批量落库 |
>
> **重点讲消息队列削峰**：
> 50 万 QPS 的点赞不需要每条都实时写库。流程改为：
> 1. 请求进来，先在 Redis 中 `ZINCRBY` 更新计数（微秒级响应）
> 2. 同时把点赞事件发到 Kafka
> 3. 后端消费者从 Kafka 拉取，**批量聚合**后写 MongoDB（比如每 100 条或每 1 秒写一次）
> 4. 数据库压力从 50 万/秒降到几千/秒
>
> **面试要点**：遇到"扛不住怎么办"的问题，思路是**接入层 → 应用层 → 缓存层 → 数据库层 → 异步层**，逐层分析瓶颈和解决方案，不要只说"加机器"。

### Q1：为什么在服务端跑物理引擎

**问题**：物理模拟一般是客户端做的，为什么服务器上也要跑？

**你的原始回答**：为了反作弊，防止客户端被修改。

**优化后的回答**：

> 核心原因是**服务端权威（Server Authority）**，这是多人游戏的基本原则：
>
> 1. **反作弊**：射线检测（判断子弹是否命中）如果只信客户端，外挂可以实现穿墙打人、自动瞄准等。服务端必须用同一套物理引擎独立验证
> 2. **一致性**：多个客户端各自计算物理，因为浮点精度、帧率差异等原因，结果可能不一致。服务端统一仲裁
> 3. **具体场景**：碰撞检测、射线检测（枪械命中判定）、触发器（进入某个区域触发剧情/机关）
>
> 客户端仍然跑一份物理做**预测和表现**（让画面流畅），但最终结果以服务端计算为准。这叫"客户端预测 + 服务端校验"模型。

---

### Q2：CGO vs 独立进程

**问题**：为什么用 CGO 而不是把 PhysX 做成独立进程通过 RPC 通信？

**你的原始回答**：同进程调用比 RPC 时效性好。CGO 是 Go 原生支持的。CGO 调用有成本，需要注意内存管理。

**优化后的回答**：

> **选 CGO 的核心原因是延迟**：
> - CGO 同进程调用：**纳秒级**
> - 本地 RPC（Unix Socket）：**微秒级**（多了序列化、系统调用、上下文切换的开销）
> - 物理引擎每帧要调用很多次（创建射线、查询碰撞、移动角色等），乘以调用次数后延迟差距很大
>
> **CGO 的代价**：
> - 不是免费的，每次调用约 **100~200 纳秒**（普通 Go 函数调用只需几纳秒），慢约两个数量级
> - 开销来源有三个：
>   1. **栈切换**：goroutine 使用自己的小栈，但 C 函数需要系统线程栈，每次 CGO 调用都要切换
>   2. **GC 协作**：Go 的 GC 需要暂停所有 goroutine，但管不到正在执行 C 代码的 goroutine，所以 CGO 调用前后需要额外的标记和同步
>   3. **调度器锁定**：goroutine 进入 C 代码后，占用的系统线程被锁住，不能调度其他 goroutine。大量 CGO 并发调用可能导致线程数暴涨
>
> **优化策略**：减少 CGO 调用次数。尽量把多个操作合并成一次 C 调用（比如批量查询），而不是频繁地逐个跨语言交互。

### Q3：C++ 回调 Go 的机制

**问题**：触发器碰撞后，C++ 怎么通知 Go 来处理业务逻辑？

**你的原始回答**：Go 端 export 一个回调函数，用 map 存储所有触发器的回调。通过 register 函数注册给 C 使用。

**优化后的回答**：

> 整体流程分三步：
>
> **第一步：Go 侧导出函数给 C 调用**
> 使用 `//export goTriggerCallback` 注解，Go 编译器会生成一个 C 可调用的函数符号。这个函数是全局唯一的入口。
>
> **第二步：注册具体的回调**
> 每个触发器创建后，调用 `RegisterTriggerCallback(trigger, callbackFunc)` 把具体的处理函数存到一个 Go 侧的全局 map 中（key 是触发器的指针/ID，value 是回调函数）。
>
> **第三步：触发时的调用链**
> ```
> PhysX 检测到碰撞 → C++ onTrigger() → 调用 Go 导出的 goTriggerCallback()
> → Go 根据触发器指针从 map 中找到对应的回调函数 → 执行业务逻辑
> ```
>
> **并发安全问题**：这个 map 存在并发读写风险——注册回调时在写 map，物理 Step 回调时在读 map。Go 的普通 map 并发读写会直接 panic。
>
> 解决方案：
> - **`sync.RWMutex`**（推荐）：注册时加写锁，回调时加读锁。读多写少场景下性能好，多个触发器回调可以并行读
> - **`sync.Map`**：标准库提供的并发安全 map，适合 key 稳定、读多写少的场景

### Q4：内存管理 — GC 管不了 C++ 的内存

**问题**：C++ 侧分配的物理对象，Go 的 GC 能回收吗？怎么防止内存泄漏？

**你的原始回答**：GC 回收不了，要手动释放。没有兜底机制的思路。

**优化后的回答**：

> Go 的 GC 只管 Go 堆上分配的内存，**C/C++ 通过 malloc/new 分配的内存，GC 完全不知道它们的存在**。如果不手动释放，就是内存泄漏。
>
> **两层防御策略**：
>
> **第一层：主动释放（主要手段）**
> - 对于生命周期在函数内的对象，用 `defer` 确保释放：
>   ```go
>   actor := physx.CreateSphere(pos, radius)
>   defer physx.DestroyActor(actor)
>   // 即使中间 panic 了，defer 也会执行
>   ```
> - 对于长生命周期的对象（如 NPC 的物理体），在对象销毁的业务逻辑中显式调用释放函数（比如 NPC 死亡时调 DestroyActor）
>
> **第二层：Finalizer 兜底（防御性编程）**
> - `runtime.SetFinalizer` 可以给 Go 对象注册一个析构回调——当 GC 回收 Go 侧的 Actor 对象时，自动释放 C++ 内存：
>   ```go
>   func CreateSphere(pos Vec3f, radius float32) *Actor {
>       actor := &Actor{Handle: C.createSphere(...)}
>       runtime.SetFinalizer(actor, func(a *Actor) {
>           C.destroyRigidBody(a.Handle)
>       })
>       return actor
>   }
>   ```
> - **注意**：Finalizer 不保证及时执行（可能延迟数秒甚至更久），所以它只是兜底，不能替代主动释放
>
> **第三层：场景级批量清理**
> - DestroyScene 时遍历所有已注册的物理对象，统一释放。即使个别对象漏掉了，场景销毁时也能兜住

### Q5：物理计算超时怎么办

**问题**：场景里 200 个玩家 + 大量 NPC，物理 Step 超过 33ms 怎么办？

**你的原始回答**：让客户端去渲染。

**问题分析**：
- 服务端本来就不渲染，问题不在渲染而在物理计算本身太慢

**优化后的回答**：

> 遇到"系统扛不住"的问题，思路是：**先减少计算量 → 再并行化 → 最后异步化**。
>
> **1. 减少计算量（最优先）**
> - **物理 LOD**：离玩家远的 NPC 降低物理精度或跳过物理模拟（类似图形 LOD 的思路）
> - **简化碰撞体**：用球体、胶囊体代替复杂网格碰撞体，计算量差距是数量级的
> - **休眠机制**：PhysX 自带 sleep 功能，静止不动的物体自动休眠，不参与计算
>
> **2. 分区并行化**
> - 把大场景拆成多个物理子场景（比如按区域划分），每个区域是一个独立的 PhysX Scene
> - 不同区域可以分配到不同线程甚至不同进程去并行计算
> - PhysX 本身支持多线程 Step
>
> **3. 异步化**
> - 把物理 Step 放到独立的 goroutine 里异步执行
> - 主游戏循环用上一帧的物理结果，允许一帧的延迟
> - 当帧的业务逻辑和下帧的物理模拟可以并行
>
> **4. 监控告警**
> - 每帧记录 Step 的耗时，超过阈值（比如 20ms）打告警日志
> - 便于发现性能退化和定位问题

### Q6：跨平台适配的坑

**问题**：Windows 和 Linux 跨平台适配遇到过什么坑？

**你的原始回答**：Linux 用 GCC 编译，Windows 用 MSVC 编译，生成的库不跨平台。CGO 底层用 GCC 所以 Linux 没问题。Windows 下客户端导出的场景不能直接在 Linux 上用，需要转换。

**优化后的回答**：

> 主要踩了两个坑：
>
> **坑1：编译工具链不兼容**
> - CGO 底层使用 GCC/Clang，但 PhysX 在 Windows 上官方只支持 MSVC 编译
> - MSVC 和 GCC 的 ABI（二进制接口）不兼容：name mangling 规则不同、异常处理机制不同、STL 内存布局不同
> - **解决方案**：不走 CGO 直接链接。Windows 上把 PhysX 封装成 DLL，导出纯 C 接口（`extern "C"` + `__declspec(dllexport)`）。Go 侧用不同的 build tag（`//go:build linux` / `//go:build windows`）区分平台实现
>
> **坑2：PhysX 序列化格式不跨平台**
> - PhysX 的二进制序列化格式（.pxb）是平台相关的——Windows 上序列化的文件不能直接在 Linux 上反序列化
> - 原因：字节序、指针大小、内存对齐方式可能不同
> - **解决方案**：使用 PhysX 提供的 `PxBinaryConverter` 工具，在 Windows 端导出场景后，先转换成 Linux 格式再部署到服务器。我在 Go 侧封装了 `ConvertSceneToLinux()` 函数自动完成转换
>
> **架构层面的处理**：
> - Go 代码用 build tag 做平台隔离，共用同一套上层接口（type.go、object.go），底层实现各写一份
> - Windows 版用 ID 管理物理对象（int 映射），Linux 版用指针直接操作——因为 DLL 跨进程传指针不安全，用 ID 更稳妥

## 3. 回答技巧

- **遇到不会的不要直接说"不知道"**。先说思考方向："我觉得方向上应该是xxx，但具体细节我还需要再深入研究"。给出方向比沉默强10倍
- **技术选型题一定要对比回答**："我选了 A 而不是 B，因为在 xxx 场景下 A 的优势是 xxx"
- **设计题要分层回答**：不管是扩容还是优化，都按"接入层 → 应用层 → 缓存层 → 数据库层 → 异步层"展开
- **数字要敏感**：MongoDB 16MB 文档限制、CGO 调用 100~200ns、Redis 单线程 10万+ QPS、网络延迟量级（本机微秒、同机房毫秒）
- **区分"我做的"和"我了解的"**：面试官最反感的是把别人的工作说成自己的，然后问深了答不上来。如果是领导决定的技术方案，说"团队讨论后决定用 xxx，我的理解是 xxx"
